# 기술문서 작성 안내

본 문서는 Microsoft 공식 문서 스타일을 참고하여 기술문서를 정확하고, 명확하고, 일관되게 작성하기 위한 원칙과 방법을 정리한 안내 문서이다.  
목적은 독자가 필요한 정보를 빠르게 찾고, 정확하게 이해하고, 실제 작업에 바로 적용할 수 있도록 돕는 데 있다.

이 안내는 설치 문서, 설정 문서, 운영 문서, 장애 대응 문서, 기능 설명 문서, API 문서, 개발자 문서, 사용자 안내 문서, 매뉴얼, 위키 문서처럼 실무에서 사용하는 기술문서 전반에 적용한다.  
문서 유형에 따라 구조와 강조점은 달라질 수 있지만, 독자 중심 구성, 명확한 표현, 일관된 용어, 수행 가능한 설명이라는 기본 원칙은 공통으로 적용한다.

## 1. 목적

기술문서는 정보를 나열하는 글이 아니다.  
기술문서는 독자가 필요한 내용을 이해하고, 필요한 작업을 올바르게 수행하도록 돕는 글이다.

따라서 기술문서는 다음 기준을 충족해야 한다.

- 필요한 정보를 빠르게 찾을 수 있어야 한다.
- 문장의 뜻이 한 가지로만 이해되어야 한다.
- 설명을 실제 작업에 바로 적용할 수 있어야 한다.
- 문서 전체의 구조와 표현이 일관되어야 한다.

## 2. 적용 대상

본 안내는 다음 문서에 적용한다.

- 설치 문서
- 설정 문서
- 운영 문서
- 장애 대응 문서
- 기능 설명 문서
- API 문서
- 개발자 문서
- 사용자 안내 문서
- 매뉴얼
- 위키 문서

문서 유형에 따라 구성 방식은 달라질 수 있다.  
예를 들어 설치 문서는 절차와 확인 방법이 중심이 되고, 참조 문서는 항목 정의와 제약 조건이 중심이 된다.  
다만 어떤 문서이든 독자가 필요한 정보를 빠르게 찾고, 한 가지 뜻으로 이해하고, 실제 작업에 적용할 수 있어야 한다는 기준은 동일하다.

## 3. 작성 전 정리할 사항

문서를 쓰기 전에 아래 사항을 먼저 정한다.  
이 단계가 정리되어야 설명 수준, 문서 구조, 예시의 범위를 안정적으로 정할 수 있다.

### 3.1 대상 독자

누가 이 문서를 읽는지 먼저 정한다.  
초급 사용자, 일반 사용자, 운영 담당자, 개발자, 관리자처럼 독자를 구체적으로 구분한다.

예:

- 초급 사용자를 위한 문서라면 개념 설명과 화면 예시를 더 많이 넣는다.
- 운영 담당자를 위한 문서라면 절차, 점검 항목, 장애 기준을 더 분명하게 쓴다.
- 개발자를 위한 문서라면 설정 키, API 필드, 예외 처리 조건을 더 구체적으로 적는다.

### 3.2 문서 목적

독자가 왜 이 문서를 읽는지 정한다.  
설치하려는 것인지, 설정을 바꾸려는 것인지, 문제를 해결하려는 것인지에 따라 문서의 구성과 설명 방식이 달라진다.

문서 목적은 보통 다음처럼 나눌 수 있다.

- 무언가를 설치한다.
- 설정 값을 이해하거나 변경한다.
- 기능 동작 방식을 이해한다.
- 오류 원인을 찾고 해결한다.

### 3.3 기대 결과

문서를 읽은 뒤 독자가 무엇을 할 수 있어야 하는지 정한다.  
기대 결과는 가능한 한 행동으로 표현한다.

예:

- 에이전트를 설치할 수 있다.
- 설정 파일을 수정할 수 있다.
- 서비스 상태를 확인할 수 있다.
- 오류 원인을 확인할 수 있다.

### 3.4 문서 유형

문서가 어떤 유형인지 먼저 정한다.  
문서 유형에 따라 문서의 구조와 문체가 달라진다.

대표적인 유형은 다음과 같다.

- 설치 문서
- 개념 설명 문서
- 절차 문서
- 문제 해결 문서
- 참조 문서

예를 들어 설치 문서는 단계와 확인 방법이 중요하고, 개념 설명 문서는 정의와 예시가 중요하다.  
문제 해결 문서는 증상, 원인, 확인 방법, 해결 방법의 흐름이 분명해야 한다.

### 3.5 핵심 질문

독자가 이 문서를 열었을 때 가장 먼저 묻게 될 질문을 한 문장으로 정리한다.  
핵심 질문이 분명해야 문서가 독자가 원하는 순서로 전개된다.

예:

- 에이전트를 어떻게 설치하는가
- 이 설정 값을 왜 바꿔야 하는가
- 인증 오류가 발생하면 무엇을 확인해야 하는가
- 백업 파일은 어디에서 확인하는가

## 4. 기본 원칙

### 4.1 독자 기준으로 구성한다

문서는 작성자가 설명하고 싶은 순서가 아니라, 독자가 필요로 하는 순서로 구성한다.  
독자가 먼저 궁금해할 내용, 먼저 해야 할 행동, 먼저 확인해야 할 조건을 앞에 둔다.

예를 들어 설치 문서라면 배경 설명보다 사전 조건과 설치 절차가 먼저 와야 한다.  
장애 대응 문서라면 이론 설명보다 증상 확인과 점검 순서가 먼저 와야 한다.

### 4.2 수행을 돕는 방향으로 쓴다

기술문서는 설명 자체보다 활용이 중요하다.  
독자가 실제로 해야 할 행동, 확인해야 할 조건, 판단해야 할 기준이 문서에 분명하게 드러나야 한다.

개념 설명은 필요할 때만 넣는다.  
설명은 실제 이해와 수행에 도움이 되는 범위 안에 머물러야 한다.

좋지 않은 예:

- 서비스 재시작 전에 시스템 구조와 내부 동작 원리를 먼저 설명한다.

좋은 예:

- 서비스를 다시 시작하기 전에 현재 상태를 확인한다.
- 서비스를 다시 시작한다.
- 로그를 확인해 정상적으로 시작되었는지 확인한다.

핵심:

- 설명보다 수행 순서를 먼저 보여 주어야 한다.
- 독자가 지금 해야 할 행동과 확인 기준이 문장에 드러나야 한다.

### 4.3 짧고 분명하게 쓴다

문장은 짧아야 한다.  
길고 복잡한 문장은 의미를 흐리고 오해를 만든다.  
한 문장에는 가능한 한 하나의 핵심만 담는다.

좋지 않은 예:

- 설정 파일을 수정한 뒤 필요에 따라 서비스를 다시 시작하고 로그를 확인하여 반영 여부를 점검한다.

좋은 예:

- 설정 파일을 수정한다.
- 서비스를 다시 시작한다.
- 로그를 확인해 변경 사항이 반영되었는지 확인한다.

핵심:

- 한 문장에 여러 행동을 넣지 않는다.
- 행동과 확인 기준을 짧게 나눈다.

### 4.4 한 개념에는 한 용어만 쓴다

같은 대상을 여러 이름으로 부르지 않는다.  
한 번 정한 용어는 문서 전체에서 그대로 유지한다.

예를 들어 `설정 파일`과 `구성 파일`을 같은 뜻으로 섞어 쓰지 않는다.  
둘 중 하나를 정했으면 끝까지 같은 표현을 사용한다.

같은 이유로 `배포`, `설치`, `적용`처럼 비슷해 보이는 말도 구분해서 써야 한다.  
뜻이 다르면 구분해서 쓰고, 같은 뜻이라면 하나로 통일한다.

좋지 않은 예:

- `config.yaml`은 설정 파일이다. 이 구성 파일을 수정한 뒤 변경 사항을 적용한다.

좋은 예:

- `config.yaml`은 설정 파일이다. 설정 파일을 수정한 뒤 변경 사항을 적용한다.

핵심:

- 같은 대상은 같은 이름으로만 불러야 한다.
- 독자가 용어 차이를 추측하게 만들지 말아야 한다.

### 4.5 문장의 중의성을 없앤다

기술문서의 문장은 한 가지 뜻으로만 읽혀야 한다.  
두 가지 이상으로 해석될 수 있는 문장은 고쳐 쓴다.

특히 다음 경우를 주의한다.

- 무엇을 먼저 하는지 순서가 분명하지 않은 경우
- 조건의 적용 범위가 분명하지 않은 경우
- 수식하는 대상이 분명하지 않은 경우
- 한 문장에 여러 행동이 섞여 있어 해석이 갈리는 경우

좋지 않은 예:

- `설정 파일`과 로그를 확인한 뒤 서비스를 다시 시작한다.

좋은 예:

- `설정 파일`을 확인한다.
- 로그를 확인한다.
- 서비스를 다시 시작한다.

핵심:

- 순서가 중요하면 행동을 나눈다.
- 한 문장 안에 여러 행동을 넣어 해석 여지를 만들지 않는다.

### 4.6 구조와 표현을 끝까지 유지한다

문서 전체에서 같은 수준의 제목은 같은 형식으로 쓰고, 같은 종류의 정보는 같은 방식으로 제시한다.  
예를 들어 절차 문서라면 각 단계의 문장 형식과 확인 방식이 문서 끝까지 일정해야 한다.

표기 방식도 일관되어야 한다.  
파일명, 명령어, 설정 키, 경로, 환경 변수 표기가 중간에 바뀌면 독자의 이해를 방해한다.

좋지 않은 예:

- 1단계에서는 `config.yaml`이라고 쓰고, 뒤에서는 config 파일이라고 쓴다.
- 어떤 곳에서는 **저장**이라고 쓰고, 다른 곳에서는 저장 버튼이라고만 쓴다.

좋은 예:

- 문서 전체에서 `config.yaml`로 통일한다.
- 문서 전체에서 **저장** 버튼으로 통일한다.

핵심:

- 제목 형식, 용어, 표기 방식, 단계 서술 방식은 처음부터 끝까지 같아야 한다.

## 5. 문서 구조

기술문서는 필요한 정보를 쉽게 찾을 수 있도록 구조를 분명하게 나누어야 한다.  
가능하면 아래 구조를 기본으로 삼는다.

### 5.1 기본 구조

많은 기술문서는 다음 흐름을 따르면 안정적으로 구성할 수 있다.

- 개요
- 사전 조건
- 본문
- 확인 방법
- 문제 해결
- 관련 문서

이 구조를 그대로 모두 써야 하는 것은 아니다.  
다만 독자가 문서를 읽는 흐름을 기준으로 보면, 이 순서는 대부분의 기술문서에 잘 맞는다.

### 5.2 문서 유형에 맞게 조정한다

문서 구조는 문서 유형에 따라 강조점이 달라진다.  
따라서 모든 문서가 같은 비중으로 같은 항목을 가져야 하는 것은 아니다.

예를 들면 다음과 같다.

- 설치 문서: `사전 조건`, `설치 절차`, `확인 방법`이 중요하다.
- 개념 설명 문서: `개요`, `핵심 개념`, `예시`, `관련 문서`가 중요하다.
- 문제 해결 문서: `증상`, `원인`, `확인 방법`, `해결 방법`이 중요하다.
- 참조 문서: `항목 정의`, `속성`, `제약 조건`, `예시`가 중요하다.

기본 구조는 출발점으로 삼고 실제 구성은 문서 목적에 맞게 조정한다.

### 5.3 각 구성 요소의 역할

기본 구조에 자주 포함되는 항목은 보통 다음 역할을 가진다.

- `개요`: 문서의 목적, 대상 독자, 기대 결과를 짧게 설명한다.
- `사전 조건`: 필요한 권한, 환경, 도구, 선행 지식을 정리한다.
- `본문`: 문서 목적에 맞는 내용만 넣는다.
- `확인 방법`: 정상 수행 여부를 독자가 직접 확인할 수 있도록 쓴다.
- `문제 해결`: 자주 발생하는 문제와 해결 방법을 정리한다.
- `관련 문서`: 다음 단계에서 읽어야 할 문서나 함께 봐야 할 문서를 연결한다.

예를 들어 절차 문서에서는 `본문`이 단계 중심이 되어야 하고, 개념 문서에서는 `본문`이 정의와 예시 중심이 되어야 한다.

### 5.4 문서 유형별 최소 구성 요소를 정한다

문서를 조직 단위로 운영한다면 문서 유형별 최소 구성 요소를 정해 두는 편이 좋다.  
문서를 작성하는 사람마다 구조가 크게 달라지면 품질을 일정하게 유지하기 어렵다.

예를 들어 다음 기준을 둘 수 있다.

- 설치 문서: `개요`, `사전 조건`, `설치 절차`, `확인 방법`
- 장애 대응 문서: `증상`, `영향 범위`, `확인 방법`, `조치 방법`
- 참조 문서: `항목 정의`, `값 또는 형식`, `제약 조건`, `예시`

이 기준은 문서의 목적에 맞게 조정할 수 있다.  
다만 독자가 무엇을 어디에서 찾아야 하는지 예측할 수 있도록 기본 틀은 유지하는 편이 좋다.

## 6. 제목과 소제목 작성 규칙

제목과 소제목은 문서의 길잡이 역할을 해야 한다.  
제목만 훑어도 문서의 흐름과 내용 구성이 보여야 한다.

### 6.1 제목 원칙

문서 제목은 문서의 주제와 범위를 즉시 드러내야 한다.  
막연한 제목이나 장식적인 제목은 사용하지 않는다.

좋은 제목은 다음 조건을 충족한다.

- 무엇에 대한 문서인지 바로 알 수 있다.
- 문서의 범위가 드러난다.
- 다른 문서와 함께 봐도 구분이 쉽다.
- 검색과 분류에 유리하다.

적절한 제목 예시는 다음과 같다.

- 기술문서 작성 안내
- API 인증 설정 방법
- 서비스 재시작 절차
- 로그 수집 설정 방법
- 백업 파일 복구 절차

피해야 할 제목 예시는 다음과 같다.

- 시작하기
- 참고
- 설명
- 안내
- 알아두기

### 6.2 소제목 원칙

같은 수준의 제목은 같은 형식으로 쓴다.  
같은 수준의 제목은 문법 구조도 맞춘다.  
제목 아래에는 반드시 그 제목에 해당하는 내용을 넣는다.  
추상적인 소제목은 피한다.

적절한 예시는 다음과 같다.

- 개요
- 사전 조건
- 설치 절차
- 설정 방법
- 결과 확인
- 문제 해결

적절하지 않은 예시는 다음과 같다.

- 기타
- 참고
- 설명
- 알아두기
- 시작하기

### 6.3 제목만 읽어도 흐름이 보여야 한다

좋은 문서는 제목만 훑어도 문서의 흐름이 보인다.  
독자는 처음부터 끝까지 모든 문장을 읽지 않고, 먼저 제목과 소제목을 훑어보는 경우가 많다.

예를 들어 아래 구조는 흐름이 비교적 잘 보인다.

- 개요
- 사전 조건
- 설치 절차
- 결과 확인
- 문제 해결

반면 아래 구조는 문서의 역할이 잘 드러나지 않는다.

- 설명
- 참고
- 기타
- 알아두기

## 7. 문장 작성 규칙

### 7.1 짧게 쓴다

불필요한 단어를 줄인다.  
의미를 더하지 않는 수식어를 없앤다.

좋지 않은 예:

- 다음 명령으로 서비스를 다시 시작할 수 있다.
- 설정 파일에서 포트 값을 변경해야 한다.

좋은 예:

- 다음 명령으로 서비스를 다시 시작한다.
- 설정 파일에서 포트 값을 변경한다.

핵심:

- 할 수 있다, 해야 한다처럼 불필요하게 우회하는 표현을 줄인다.
- 의미가 달라지지 않으면 더 짧은 문장을 선택한다.

### 7.2 직접 쓴다

우회 표현을 피한다.  
행동 주체와 행동 내용을 바로 드러낸다.

좋지 않은 예:

- 서비스 상태 확인이 필요하다.
- 설정이 저장되면 서비스가 재시작된다.

좋은 예:

- 서비스 상태를 확인한다.
- 설정을 저장한 뒤 서비스를 다시 시작한다.

핵심:

- 해야 할 행동을 명사형이 아니라 동사형으로 바로 쓴다.
- 수동 표현보다 실제 수행 문장을 우선한다.

### 7.3 구체적으로 쓴다

넓은 표현보다 좁고 분명한 표현을 쓴다.  
가능하면 조건, 수치, 위치, 파일명, 명령어를 함께 쓴다.

좋지 않은 예:

- 오류가 발생하면 로그를 확인한다.
- 환경 변수를 설정한다.

좋은 예:

- 서비스가 시작되지 않으면 `/var/log/app.log` 파일을 확인한다.
- 환경 변수 `API_KEY`에 발급받은 키를 설정한다.

핵심:

- 오류, 값, 위치를 구체적으로 적는다.
- 독자가 무엇을 봐야 하는지 바로 알 수 있어야 한다.

### 7.4 기준을 함께 쓴다

기준이 없는 표현은 피한다.  
독자가 무엇을 보고 어떻게 판단해야 하는지 문장 안에서 알 수 있어야 한다.

좋지 않은 예:

- 서비스가 정상적으로 실행되는지 확인한다.
- 필요한 경우 포트를 변경한다.

좋은 예:

- `systemctl status myapp` 명령으로 서비스 상태가 `active (running)`인지 확인한다.
- 포트 `8080`이 이미 사용 중이면 `server.port` 값을 다른 포트로 변경한다.

핵심:

- 확인 기준은 결과 상태로 적는다.
- 조건이 있다면 그 조건을 문장 안에 드러낸다.

### 7.5 실제 동작이 드러나는 동사를 쓴다

뜻이 넓은 동사는 문장을 흐리게 만든다.  
가능하면 실제 동작이 보이는 동사를 쓴다.

좋지 않은 예:

- 로그를 처리한다.
- 설정을 관리한다.
- 점검을 수행한다.

좋은 예:

- 로그를 수집한다.
- 설정 값을 수정한다.
- 서비스 상태를 확인한다.

핵심:

- 추상 동사보다 실제 동작 동사를 쓴다.
- 독자가 행동을 바로 떠올릴 수 있어야 한다.

## 8. 용어와 표기 규칙

### 8.1 쉬운 한국어를 먼저 쓴다

기술문서는 가능한 한 쉬운 한국어로 쓴다.  
쉬운 한국어로도 정확하게 표현할 수 있다면 어려운 한자어, 관성적인 전문 용어, 불필요한 외래어를 줄이는 편이 좋다.

좋지 않은 예:

- 인증을 수행하기 위해 적절한 자격 증명을 활용한다.
- 장애 발생 시 관련 리소스를 점검한다.
- 사용자는 적절한 파라미터를 인풋해야 한다.

좋은 예:

- 인증에 필요한 계정 정보를 입력한다.
- 장애가 발생하면 관련 서비스와 로그를 확인한다.
- 사용자는 필요한 값을 입력한다.

핵심:

- 낯선 말보다 익숙한 말을 먼저 쓴다.
- 쉬운 말로도 정확하게 쓸 수 있으면 쉬운 말을 택한다.

### 8.2 필요하면 익숙한 영문을 함께 적는다

한국어보다 영문이 더 익숙하거나, 한국어와 영문을 함께 적는 편이 더 정확한 경우에는 처음 한 번 영문을 병기한다.  
이후에는 문서 전체에서 한 가지 표현 방식을 일관되게 유지한다.

예를 들어 다음 방식이 자연스럽다.

- `매개변수(parameter)`
- `전문 용어(jargon)`
- `스크립트(script)`
- `종단점(endpoint)`

반대로 독자에게 더 익숙한 영문 개념을 억지로 낯선 번역어로만 적으면 오히려 이해를 방해할 수 있다.  
중요한 것은 한국어 우선, 필요할 때 정확하고 익숙한 영문 병기, 그리고 문서 전체의 일관성이다.

### 8.3 약어는 처음에 풀어 쓴다

약어는 첫 등장 시 전체 명칭과 함께 쓴다.  
그 뒤부터는 약어만 사용해도 된다.

예:

- Application Programming Interface(API)
- Secure Shell(SSH)

### 8.4 비슷한 말을 섞지 않는다

비슷한 말을 같은 뜻으로 섞어 쓰지 않는다.

예:

- 배포 / 설치 / 적용
- 오류 / 장애
- 설정 파일 / 구성 파일

뜻이 다르면 구분해서 쓰고, 같은 뜻이라면 하나로 통일한다.  
예를 들어 패키지를 서버에 올리는 작업은 `배포`, 프로그램을 시스템에 넣는 작업은 `설치`, 변경 사항이 실제 동작에 반영되는 상태는 `적용`처럼 구분해서 쓴다.

### 8.5 코드와 기술 요소는 백틱으로 쓴다

코드 조각, 파일명, 변수명, 함수명, 메서드명, 클래스명, 명령어, 설정 키, 환경 변수, 경로, API 필드명처럼 일반 문장과 구분해야 하는 기술 요소는 백틱으로 쓴다.  
이 요소들은 문서 전체에서 같은 방식으로 표기해야 한다.

예:

- `settings.json`
- `npm install`
- `projectName`
- `getUserById()`
- `UserService`
- `DB_HOST`
- `/var/log/app.log`
- `userId`
- `if (isEnabled)`

핵심:

- 기술 요소는 일반 설명 문장과 시각적으로 구분되어야 한다.
- 일반 개념어와 실제 식별자는 구분한다.
- 예를 들어 설정 파일이라는 일반 개념은 일반 문장으로 쓰고, 실제 파일명 `settings.json`은 백틱으로 쓴다.

### 8.6 화면에 보이는 UI 요소는 굵게 쓴다

버튼, 메뉴, 탭, 대화 상자 이름, 화면 이름, 입력란 이름처럼 사용자가 화면에서 직접 보는 요소는 굵게 쓴다.  
문장 안에서 일반 설명과 UI 이름이 섞이지 않도록 분명하게 구분해야 한다.

예:

- **저장**을 선택한다.
- **설정** > **계정**으로 이동한다.
- **프로젝트 이름** 입력란에 값을 입력한다.
- **로그인** 화면에서 계정 정보를 입력한다.

메뉴 경로를 쓸 때는 `>`를 사용해 흐름을 보여 준다.  
이때 `>` 자체는 일반 문자로 두고, 실제 UI 이름만 굵게 쓴다.

과도한 강조는 피한다.  
화면에서 직접 선택하거나 확인해야 하는 이름에만 적용하고, 같은 유형의 UI 요소에는 일관되게 적용한다.

### 8.7 UI 요소와 기술 요소를 섞어 쓰지 않는다

화면에서 보이는 이름과 코드, 파일, 설정 키는 성격이 다르다.  
같은 방식으로 표기하면 독자가 무엇이 화면 요소이고 무엇이 기술 요소인지 구분하기 어렵다.

좋지 않은 예:

- `저장`을 눌러 `설정` 메뉴로 이동한 뒤 프로젝트 이름을 수정한다.

좋은 예:

- **저장**을 선택한다.
- **설정** > **프로젝트**로 이동한다.
- `projectName` 값을 수정한다.

핵심:

- 화면에 보이는 요소는 굵게 쓴다.
- 코드, 파일명, 변수명, 함수명, 명령어, 설정 키는 백틱으로 쓴다.
- 같은 문장 안에서도 두 표기 방식을 역할에 맞게 구분한다.

### 8.8 예외 기준도 함께 정한다

용어 원칙은 일관되어야 하지만, 문서 유형에 따라 예외가 필요할 수 있다.  
예를 들어 API 참조 문서나 코드 중심 문서에서는 영문 용어를 기본 표기로 두는 편이 더 정확할 수 있다.

이 경우에도 기준은 분명해야 한다.  
어떤 문서에서 어떤 표현을 기본으로 쓰는지 먼저 정하고, 문서 안에서 그 기준을 끝까지 유지한다.

## 9. 절차 문서 작성 규칙

절차 문서는 독자가 그대로 따라 하는 문서이므로 특히 명확하게 써야 한다.

### 9.1 각 단계는 행동으로 시작한다

좋지 않은 예:

1. 포트 설정을 위한 파일을 열 수 있다.
2. 필요한 경우 값을 변경할 수 있다.
3. 이후 서비스 재시작이 필요할 수 있다.

좋은 예:

1. `config.yaml` 파일을 연다.
2. `port` 값을 `8080`으로 수정한다.
3. 서비스를 다시 시작한다.

핵심:

- 각 단계는 행동 동사로 시작한다.
- 가능성이나 권고보다 실제 수행 문장을 쓴다.

### 9.2 한 단계에는 한 핵심 행동만 넣는다

원칙적으로 한 단계에는 한 가지 핵심 행동만 넣는다.  
행동이 둘 이상이면 단계를 나눈다.  
한 단계 안에 여러 행동을 묶으면 따라 하기 어렵다.

다만 같은 위치에서 연속으로 수행하는 단순한 동작은 한 단계로 묶을 수 있다.  
이때도 문장은 짧고 분명해야 한다.

좋지 않은 예:

1. `config.yaml` 파일을 열고 `port` 값을 수정한 뒤 저장하고 서비스를 다시 시작한다.

좋은 예:

1. `config.yaml` 파일을 연다.
2. `port` 값을 수정한다.
3. 파일을 저장한다.
4. 서비스를 다시 시작한다.

허용 가능한 예:

1. **설정** 화면에서 **고급 옵션**을 열고 **로그 수집 사용**을 선택한다.

핵심:

- 원칙은 한 단계에 한 핵심 행동이다.
- 같은 위치의 단순한 연속 동작만 예외적으로 묶는다.

### 9.3 순서가 중요하면 번호를 쓴다

순차 실행이 필요한 절차는 번호 목록으로 작성한다.  
순서와 관계없는 항목만 글머리표를 사용한다.

예를 들어 설치 절차, 복구 절차, 점검 순서는 번호 목록이 적절하다.  
반면 준비물, 주의사항, 선택 가능한 옵션은 글머리표가 더 적절하다.

### 9.4 특정 입력 방식에 의존하지 않는 표현을 우선한다

특정 입력 방식이 중요하지 않으면 마우스, 터치, 키보드 같은 입력 방식을 드러내지 않는 중립적인 표현을 먼저 쓴다.  
기본 동사는 `선택한다`를 사용한다.  
이는 다양한 사용 환경과 접근성을 함께 고려하기 위한 원칙이다.

좋은 예:

- **확인**을 선택한다.
- **다음**을 선택한다.
- **설정** > **보안**으로 이동한다.

예외:

- 항목을 마우스 오른쪽 단추로 클릭한다.
- 막대를 끌어 원하는 위치로 이동한다.

핵심:

- 입력 방식이 중요하지 않으면 `선택한다`를 기본으로 쓴다.
- 우클릭, 드래그처럼 특정 동작이 기술적으로 필수일 때만 해당 용어를 정확하게 적는다.

### 9.5 단계마다 확인 기준을 둘 수 있으면 함께 쓴다

절차가 길거나 중간 실패 가능성이 큰 작업은 단계별 확인 기준을 두는 편이 좋다.  
독자는 다음 단계로 넘어가기 전에 현재 단계가 정상적으로 끝났는지 확인할 수 있어야 한다.

좋지 않은 예:

1. 패키지를 설치한다.
2. 서비스를 시작한다.
3. 다음 단계로 진행한다.

좋은 예:

1. 패키지를 설치한다.
2. 설치 로그에 오류가 없는지 확인한다.
3. 서비스를 시작한다.
4. `systemctl status myapp` 명령으로 서비스 상태가 `active (running)`인지 확인한다.

핵심:

- 절차 중간에 확인 기준을 넣으면 실패 지점을 빨리 찾을 수 있다.
- 다음 단계로 넘어가기 전에 성공 여부를 판단할 수 있어야 한다.

## 10. 예시, 목록, 표, 링크 작성 규칙

### 10.1 예시는 실제 수행에 도움이 되어야 한다

예시는 설명을 보완하는 것이 아니라, 설명을 실제로 보여 주는 역할을 해야 한다.

다음 원칙을 따른다.

- 예시는 짧게 쓴다.
- 예시는 실행 가능해야 한다.
- 예시 앞에는 무엇을 보여 주는지 짧게 설명한다.
- 예시 뒤에는 기대 결과나 확인 방법을 적는다.
- 변수명, 파일명, 함수명은 의미 있게 쓴다.

예:

다음 명령은 서비스 상태를 확인하는 방법을 보여 준다.

`systemctl status myapp`

명령을 실행한 뒤 상태가 `active (running)`으로 표시되는지 확인한다.

### 10.2 목록은 같은 형태로 맞춘다

같은 목록 안의 항목은 문장 구조를 맞춘다.

좋지 않은 예:

- 설정 파일 열기
- 값을 수정한다.
- 서비스 재시작 방법

좋은 예:

- 설정 파일을 연다.
- 값을 수정한다.
- 서비스를 재시작한다.

핵심:

- 같은 목록 안에서는 품사와 문장 구조를 맞춘다.
- 명사형, 동사형, 설명형을 섞지 않는다.

### 10.3 표는 비교할 때만 쓴다

표는 값, 속성, 옵션, 차이점을 비교할 때 사용한다.  
설명이 중심이면 표보다 문단이 더 적절하다.

예를 들어 포트 번호, 기본값, 설명을 함께 보여 줄 때는 표가 적절하다.  
반대로 설정 변경 이유를 설명할 때는 문단이 더 적절하다.

### 10.4 링크 문구는 목적지를 드러내야 한다

링크 문구만 읽어도 어디로 이동하는지 알 수 있어야 한다.

좋지 않은 예:

- 자세한 내용은 여기
- 이 문서를 참고한다

좋은 예:

- 인증 설정 방법은 API 인증 구성 문서를 참고한다.
- 로그 분석 절차는 장애 대응 가이드를 참고한다.

핵심:

- 링크 문구만 읽어도 이동 목적을 알 수 있어야 한다.
- 여기, 참고 같은 막연한 문구는 피한다.

### 10.5 관련 문서는 다음 행동 기준으로 연결한다

관련 문서를 나열할 때는 단순히 많이 연결하는 것보다, 독자가 다음에 무엇을 해야 하는지 기준으로 연결하는 편이 좋다.

좋지 않은 예:

- 관련 문서:
  - 설정 문서
  - 인증 문서
  - 로그 문서

좋은 예:

- 설치를 마친 뒤 설정을 계속하려면 설정 문서를 참고한다.
- 인증 오류를 해결하려면 인증 문제 해결 문서를 참고한다.

핵심:

- 관련 문서는 단순 목록보다 다음 행동과 함께 연결하는 편이 좋다.
- 문서 간 흐름을 보여 주어야 한다.

## 11. 접근성과 번역 가능성

기술문서는 더 많은 독자가 같은 뜻으로 이해할 수 있어야 한다.  
따라서 접근성과 번역 가능성도 함께 고려한다.

### 11.1 스크린샷만으로 핵심 정보를 전달하지 않는다

스크린샷은 보조 수단으로 사용한다.  
핵심 단계, 조건, 값, 경로, 명령어는 본문에도 텍스트로 적는다.

좋지 않은 예:

- 아래 화면과 같이 설정한다.

좋은 예:

- **설정** > **보안**으로 이동한다.
- `AllowedOrigins` 값을 `https://example.com`으로 설정한다.
- 필요하면 아래 스크린샷을 함께 참고한다.

### 11.2 색상만으로 상태를 구분하지 않는다

정상, 경고, 오류 같은 상태를 색상만으로 구분하면 일부 독자는 정보를 정확하게 이해하기 어렵다.  
색상과 함께 텍스트 기준도 제공한다.

좋지 않은 예:

- 빨간 항목을 수정한다.

좋은 예:

- 상태가 **오류**로 표시된 항목을 수정한다.
- `Status` 값이 `Failed`인 항목을 확인한다.

### 11.3 번역에 약한 표현을 피한다

관용 표현, 줄임말, 문화권 의존 표현, 말장난은 번역 과정에서 의미가 흔들리기 쉽다.  
가능하면 직접적이고 중립적인 표현을 사용한다.

좋지 않은 예:

- 이 설정은 꽤 까다로운 편이다.
- 버튼만 한 번 때리면 된다.

좋은 예:

- 이 설정은 입력 값의 형식을 정확히 맞춰야 한다.
- **실행**을 선택한다.

## 12. 어조

기술문서는 차분하고 단정하게 쓴다.  
과장하지 않고, 장식하지 않고, 감정적으로 밀어붙이지 않는다.

좋은 문체는 다음과 같다.

- 직접적이다.
- 침착하다.
- 판단 기준이 분명하다.
- 불필요한 감탄이나 평가가 없다.

피해야 할 문체는 다음과 같다.

- 과장하는 문체
- 추상적인 문체
- 장식적인 문체
- 근거 없이 단정하는 문체

좋지 않은 예:

- 이 기능은 매우 강력하고 아주 편리하게 사용할 수 있다.

좋은 예:

- 이 기능은 로그를 시간순으로 조회할 수 있다.

핵심:

- 평가보다 사실을 쓴다.
- 기능을 설명할 때는 실제 동작과 결과를 적는다.

## 13. 피해야 할 표현

기술문서에서는 실제 동작, 조건, 값, 위치가 드러나는 표현을 써야 한다.  
따라서 의미가 흐리거나 기준이 불분명한 표현은 피한다.

### 13.1 동사의 뜻이 지나치게 넓은 표현

다음 표현은 실제 동작이 무엇인지 분명하지 않다.  
실무 문서에서는 추상적인 동사보다 실제 동작이 드러나는 동사를 사용한다.

- 로그를 처리한다
- 배포를 진행한다
- 설정을 관리한다
- 점검을 수행한다

다음처럼 고쳐 쓴다.

- 로그를 수집한다
- 애플리케이션을 배포한다
- 설정 값을 수정한다
- 서비스 상태를 확인한다

### 13.2 기준이 드러나지 않는 표현

다음 표현은 조건이나 기준이 분명하지 않다.

- 서비스가 정상적으로 실행되는지 확인한다
- 필요한 경우 포트를 변경한다
- 문제가 발생하면 로그를 확인한다
- 올바른 값을 입력한다

다음처럼 고쳐 쓴다.

- `systemctl status myapp` 명령으로 서비스 상태가 `active (running)`인지 확인한다
- 포트 `8080`이 이미 사용 중이면 `server.port` 값을 다른 포트로 변경한다
- 인증 오류가 발생하면 `/var/log/app.log` 파일에서 `401` 오류를 확인한다
- `DB_HOST` 변수에 데이터베이스 서버 주소를 입력한다

### 13.3 근거가 없는 평가 표현

다음 표현은 기능 설명보다 홍보 문구에 가깝다.

- 강력한 보안 기능을 제공한다
- 직관적인 관리 화면을 제공한다
- 효율적인 로그 분석이 가능하다
- 손쉽게 배포할 수 있다

다음처럼 고쳐 쓴다.

- 토큰 기반 인증을 지원한다
- 관리 화면에서 사용자 권한을 변경할 수 있다
- 로그를 시간순으로 조회할 수 있다
- `deploy.sh` 스크립트로 애플리케이션을 배포한다

### 13.4 범위가 흐린 표현

다음 표현은 적용 범위가 분명하지 않다.

- 이 설정은 보안에 중요하다
- 일부 환경에서는 다르게 동작할 수 있다
- 필요한 권한이 있어야 한다

다음처럼 고쳐 쓴다.

- 이 설정은 외부 접근을 허용하므로 운영 환경에서는 변경 전에 검토해야 한다
- Windows 환경에서는 경로 구분자가 다르므로 예시 명령을 그대로 사용할 수 없다
- 이 작업을 수행하려면 관리자 권한이 필요하다

## 14. 점검표

문서를 마무리하기 전에 아래 항목을 확인한다.

1. 제목만 보고 문서 목적을 알 수 있는가
2. 대상 독자와 기대 결과가 처음에 드러나는가
3. 소제목만 훑어도 흐름이 보이는가
4. 같은 개념에 같은 용어만 쓰는가
5. 문장이 두 가지 뜻으로 읽히지 않는가
6. 기준 없는 표현을 조건이나 값으로 바꾸었는가
7. 동사의 의미가 실제 동작을 드러내는가
8. 절차 문서의 각 단계가 행동으로 시작하는가
9. 예시가 실제 수행에 도움이 되는가
10. 링크 문구만 읽어도 목적지를 알 수 있는가
11. 화면에 보이는 UI 요소를 굵게 표기했는가
12. 코드, 파일명, 변수명, 함수명, 명령어, 설정 키, 경로, 환경 변수를 백틱으로 표기했는가
13. 문서 유형에 맞는 최소 구성 요소를 갖추었는가
14. 스크린샷, 색상, 표현 방식이 접근성과 번역 가능성을 해치지 않는가
15. 예외가 필요한 경우 그 기준을 문서 안에서 분명하게 밝혔는가

## 15. 결론

기술문서는 정보를 많이 담는 문서가 아니라, 독자가 필요한 내용을 빠르게 찾고 정확하게 이해한 뒤 실제 작업에 바로 적용할 수 있게 만드는 문서이다.

좋은 기술문서는 표현을 꾸미는 데서 완성되지 않는다.  
독자가 덜 헤매고, 덜 추측하고, 더 정확하게 판단하고 수행할 수 있도록 구성될 때 비로소 제 역할을 한다.

따라서 기술문서는 작성자가 설명하고 싶은 순서보다 독자가 필요로 하는 순서로 구성해야 한다.  
문장은 한 가지 뜻으로만 읽혀야 하며, 표현은 실제 동작과 판단 기준이 드러나야 한다.  
또한 문서 전체에서 구조, 용어, 표기, 설명 방식은 일관되게 유지되어야 한다.

결국 좋은 기술문서는 많이 설명한 문서가 아니라, 필요한 정보를 찾기 쉽고, 이해하기 쉽고, 바로 사용할 수 있게 만든 문서이다.  
기술문서의 품질은 설명의 양이 아니라 정확도, 명확성, 일관성, 그리고 적용 가능성으로 결정된다.
